From client/utils/api.ts (consumer expectations):

- getMyLeaderboardPosition(period) -> GET /leaderboard/me with auth token

From docs/backend_handoff_for_leaderboard_and_categories.md (required contracts):
- GET /markets/categories -> string[] uppercase keys
- GET /leaderboard -> query: period(required), page(optional 0-index), limit(optional)
- GET /leaderboard/me -> query: period(required), auth required

Required status semantics:
- 400 for invalid period on both leaderboard endpoints
- 400 for invalid page/limit on GET /leaderboard
- 401 for missing/invalid auth on GET /leaderboard/me
- 404 for unranked authenticated user on GET /leaderboard/me
</interfaces>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> public leaderboard API | Untrusted query input crosses into backend validation and data access |
| client -> authenticated /leaderboard/me | Bearer token and period input cross into auth + profile/ranking lookup |
| backend service -> Supabase | Query construction and selected fields cross into persistence boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | T | GET /leaderboard query parsing | mitigate | Validate and bound period/page/limit before service call; return 400 on invalid values |
| T-02-02 | E | GET /leaderboard/me route | mitigate | Apply requireAuth middleware only on /me; derive user id from verified token |
| T-02-03 | I | Leaderboard service select payload | mitigate | Explicitly map response fields and avoid broad sensitive column selection |
| T-02-04 | D | Pagination limit abuse | mitigate | Enforce limit <= 100 and page >= 0; add contract tests for abuse attempts |
| T-02-05 | R | Rollback execution | accept | Add rollback runbook with command log expectations and smoke-test evidence capture |
</threat_model>
